**Purpose.** Input artifact for the future `G-RLS-MIGRATION` PR. Lists every model whose `__tablename__` declaration sits within ~2 KB of a `tenant_id` column or a `TenantMixin` reference.

**Scope of the PR that ships this file (ERR-2 Phase 3).** *Not* enabling RLS on these tables. Only fixing the actual cross-tenant leak documented in UAT Issue #3 — which was a missing tenant row at registration + a missing `tenant_id` JWT claim, *not* a missing DB-layer RLS enforcement. The app-layer tenant guard (`TenantContextMiddleware` + `attach_tenant_guard()`) was already wired before this PR and works correctly once the JWT carries a tenant id.

## Recommended G-RLS-MIGRATION rollout

Per `CLAUDE.md` G-A3.1 (the prior production incident from a bulk auto-generated migration), enabling RLS on every table at once is exactly the wrong shape of change.

1. **Pilot wave** — one low-traffic table (e.g. `pilot_branches`).
   Apply RLS in a hand-written, idempotent migration with explicit
   `IF NOT EXISTS` guards. Verify in staging for ~1 week.
2. **Critical financial wave** — invoices, journal entries, GL (~10 tables). Separate PR, separate staging window.
3. **Master-data wave** — customers, vendors, products (~20 tables).
4. **Operational wave** — the remaining ~140 tables in logical batches of 20-30.

Each wave: one migration, one PR, staging verification, gradual
rollout via a `RLS_ENFORCE_<wave>` env flag. Defense-in-depth on top of the app-layer guard — never the only line of defense.

## Prerequisite

Tickets `G-CI-DOCKER-POSTGRES` (testcontainers-postgres in CI) and `G-LEGACY-TENANT-MIGRATION` (backfill script for users still on the shared `06892550-…` tenant) must land before G-RLS-MIGRATION.
